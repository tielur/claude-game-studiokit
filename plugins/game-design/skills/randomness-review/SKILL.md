Evaluate the user's game design or mechanic for its use of randomness. Read the framework in `references/framework.md` for the full evaluation model, and check `references/gotchas.md` for common analysis mistakes to avoid.

The core idea: randomness splits into input (before the player decides) and output (after the player decides). Both have valid uses, but they create very different player experiences. Input randomness supports strategy; output randomness tests risk management but can feel unfair.

For each relevant dimension, assess what's working and what's not. Be direct and specific. End with the 1-2 highest-leverage changes.

Judge the design on its own terms. Not every game needs the same relationship with randomness.
